# ADR-0001: 数据存储方案选型

## 状态

**已接受**

## 上下文

MindFlow 是一款 Offline-First 的个人知识管理笔记应用，核心约束包括：

- **Offline-First**：离线状态下全部核心功能（新建、编辑、搜索、删除）必须可用
- **跨设备同步**：Web/iOS/Android 三端共享同一套数据，通过云端中介同步
- **单用户 10000+ 条笔记**，单品笔记上限 50000 字
- **P95 同步延迟 < 3s**
- **各端原生开发**，不采用跨平台方案
- **MVP 阶段不考虑团队协作**

我们需要解决以下存储层面的关键问题：

1. **本地真理源**：离线状态下的数据存储在哪里？以什么格式？如何保证可靠性？
2. **云端职责**：云端存储什么数据？承担什么角色（备份/中介/真理源）？
3. **搜索索引**：全文搜索（FTS5）的索引存储位置和更新策略？
4. **附件存储**：笔记中的图片等附件存储在哪？
5. **分端差异**：Web/iOS/Android 三端在存储层能否统一方案？各端差异在哪？

## 决策

### 决策 1：本地 SQLite 作为真理源（Source of Truth）

**所有端（Web/iOS/Android）均使用 SQLite 作为本地数据库**，作为数据的第一真理源。

- 读操作完全基于本地 SQLite，不等待网络
- 写操作先写入本地 SQLite，再通过同步引擎推送到云端
- 云端数据故障不影响本地数据完整性和读写能力
- 本地 SQLite 即权威数据，云端是副本/中介

### 决策 2：全文搜索使用同一 SQLite 实例的 FTS5 扩展

- 不引入独立的搜索引擎（如 Elasticsearch、Meilisearch）
- 使用 SQLite FTS5 虚拟表实现全文搜索
- 搜索索引与业务数据在同库但分离存储（FTS5 虚拟表）
- 优势：零额外运维成本、事务一致性保证、初始化零等待

### 决策 3：云端作为变更日志的中介

- 云端不直接作为真理源，而是存储所有设备的变更日志
- 云端职责：(1) 存储用户的全部 ChangeEntry，(2) 作为各端同步的中转站，(3) 提供版本历史回复，(4) 提供备份
- 云端不执行搜索计算（搜索完全在本地完成）
- 云端不执行业务逻辑（业务逻辑在客户端 Domain 层完成）
- 云端存储的笔记快照用于：新设备首次同步、版本历史、冲突还原

### 决策 4：附件存储

- MVP 阶段**不处理附件同步**：
  - 图片等附件存储在各端本地区域
  - 云端附件存储延迟到 P1/P2 阶段
  - MVP 中的图片 Markdown 语法仅本地渲染 `![alt](local_path)`
- 附件同步方案在后续版本中单独评审

### 决策 5：各端统一数据库 schema

- **三端使用完全相同的数据库 schema**（DDL 一致）
- SQL 差异通过各端的 ORM/适配器层抽象（Web: wa-sqlite, iOS: GRDB.swift, Android: Room）
- 保证数据文件的跨端兼容性（以便未来支持数据库文件直接导入导出）

## 各端存储方案细节

| 维度 | Web | iOS | Android |
|------|-----|-----|---------|
| SQLite 方案 | wa-sqlite (WASM) | GRDB.swift | Room (SQLite wrapper) |
| FTS5 支持 | wa-sqlite FTS5 扩展 | GRDB + FTS5 | Room + FTS5 |
| 数据库位置 | OPFS (Origin Private File System) | App Documents 目录 | App Internal Storage |
| 加密 | Web Crypto API + SQLite Encryption | SQLCipher / Keychain | SQLCipher / EncryptedSharedPreferences |
| 嵌入式 | WASM 约 1.5MB | 系统自带 SQLite | 系统自带 SQLite |
| 迁移工具 | 自定义 migration 脚本 | GRDB Migrations | Room Migrations |
| 数据库文件 | 单文件 `.mindflow.db` | 单文件 `MindFlow.sqlite` | 单文件 `mindflow.db` |

## 备选方案

### 备选 A：IndexedDB（取代 SQLite，Web 端）

| 对比项 | SQLite (wa-sqlite) | IndexedDB |
|--------|--------------------|-----------|
| 全文搜索 | 原生 FTS5 支持 | 不支持，需自建倒排索引 |
| 查询能力 | 完整 SQL | Key-Value / Cursor |
| 事务 | ACID | 仅单 object store 事务 |
| 多端一致性 | SQLite 三端均可原生使用 | 仅 Web 端可用 |
| 数据量 10000+ | 成熟 | 性能下降显著 |
| WASM 体积 | ~1.5MB | 原生支持 |
| **结论** | **选用** | 不满足搜索和跨端一致性需求 |

**备选 A 的否决理由**：IndexedDB 的全文搜索能力严重不足，且 Web 端的数据无法与移动端的原生 SQLite 直接兼容。

### 备选 B：Remote-First 架构（云端为主，本地缓存）

| 对比项 | Offline-First (Local Source) | Remote-First (Cloud Source) |
|--------|------------------------------|-----------------------------|
| 离线可用性 | 完全可用 | 不可用或受限 |
| 数据所有权 | 用户本地持有完整数据副本 | 数据主要存储在云端 |
| 同步复杂度 | 高（需处理冲突） | 低（云端就是真理源） |
| 首次加载速度 | 即时（本地数据） | 需等待网络 |
| **结论** | **选用** | 与 PRD 的 "Offline-First" 核心理念冲突 |

**备选 B 的否决理由**：与产品定位直接冲突。PRD 明确要求离线状态全部核心功能可用。

### 备选 C：使用云数据库服务（Supabase / Firebase）取代自建云端

这个选择与本地存储方案正交。云端服务的选择不影响本地存储决策。详细选型由后端团队在独立评审中决定。

## 后果

### 正面后果

1. **离线体验完整**：所有数据操作在本地完成，与网络状态完全解耦
2. **启动速度极快**：冷启动时无需等待任何网络请求即可展示笔记列表
3. **数据主权在用户**：用户即使停止付费/服务关闭，本地数据依然完整可用
4. **搜索延迟可预测**：FTS5 查询纯本地执行，不受网络延迟影响
5. **三端 schema 统一**：数据库 DDL 一致，跨端数据校验逻辑可复用
6. **版本历史可追溯**：ChangeEntry 日志天然记录所有变更，可回滚到任意历史点
7. **云端成本低**：云端不承担搜索计算和业务逻辑，仅需存储变更日志

### 负面后果

1. **同步层复杂度高**：需要自行实现增量同步引擎，而非依赖云数据库的实时同步能力
2. **数据冗余**：同一份笔记数据在本地和云端各存一份（本地完整副本 + 云端变更日志）
3. **FTS5 跨端性能差异**：Web 端的 WASM SQLite FTS5 性能弱于 iOS/Android 原生 SQLite（预计约 70% 性能），大容量搜索时 Web 端可能稍慢
4. **数据库迁移成本**：schema 版本升级需要三端同步发布，移动端升级滞后可能导致兼容性问题
5. **首次全量加载**：新设备首次使用时需要从云端拉取全量数据，全量同步体验需要优化（分批加载 + 进度提示）
6. **附件存储割裂**：MVP 阶段的附件仅存储在本地，无法通过其他设备查看，需要后续版本补齐

## 版本演进计划

| 阶段 | 存储能力 |
|------|---------|
| MVP (W1-W12) | 本地 SQLite + FTS5 全文搜索；云端变更日志存储；无附件同步 |
| GA (W13-W24) | 云端版本历史（30 天）；云端附件同步；数据导出（本地数据库文件导出为 Markdown+JSON） |
| v1.5+ | CRDT 字段级合并（同步增强）；附件增量同步优化；云端端到端加密 |
| v2+ | 数据库文件直接导入导出（便于迁移）；备份自动上传云端 |
