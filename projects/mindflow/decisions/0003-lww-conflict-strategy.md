# ADR-0003: LWW 冲突策略的适用性分析

## 状态

**已接受**

## 上下文

MindFlow 采用 Offline-First 架构，数据在多个设备之间通过云端中介同步。多设备编辑必然导致**数据冲突**——同一笔记在不同设备上被同时编辑，产生两个不同的版本。

冲突场景分析：

```
场景 1 — 顺序编辑（无冲突）:
  设备 A 编辑 → 保存 → 同步到云端 → 设备 B 拉取 → 设备 B 编辑
  └── 线性时间线，始终是增量更新，不会产生冲突

场景 2 — 并发编辑（字段级冲突）:
  设备 A 编辑标题 ──→ 保存 ──→ 同步云端
  设备 B 编辑正文 ──→ 保存 ──→ 同步云端
  └── 同时修改同一条笔记的不同字段，理想情况下应合并

场景 3 — 并发编辑（内容级冲突）:
  设备 A 修改正文第 1 段 ──→ 保存 ──→ 同步云端
  设备 B 修改正文第 3 段 ──→ 保存 ──→ 同步云端
  └── 同时修改同一条笔记的不同段落，理想情况下应合并

场景 4 — 离线冲突:
  设备 A 离线编辑（版本 5 → 6）
  设备 B 离线编辑（版本 5 → 6）
  设备 A 先联网同步（云端版本 = 6）
  设备 B 后联网同步（云端版本 = 6, 本地版本 = 6）
  └── 两端基于同一个基准版本各自升级，产生真正的冲突

场景 5 — 删除冲突:
  设备 A 删除笔记 → 同步云端
  设备 B 离线编辑该笔记 → 联网同步
  └── 一端删除一端编辑的处理策略
```

我们需要决定 MindFlow MVP 阶段使用什么冲突处理策略。核心权衡：

- **实现的简单性** vs **数据保留的完整性**
- **用户无感知自动合并** vs **用户参与手动解决**
- **MVP 快速交付** vs **长期演进路径**

## 决策

### 决策 1：MVP 阶段采用 LWW（Last-Writer-Wins）策略

**所有冲突一律以 "最后保存的版本胜出" 为原则自动裁决，不弹窗询问用户。**

```
LWW 裁决规则:
  1. 比较冲突双方的时间戳（timestamp 字段）
  2. 较晚的时间戳胜出
  3. 如果时间戳差异 < 1 秒（无法区分先后）：
     a. 比较 device_id（UUID 字符串字典序，确保确定性）
     b. 字典序大的 device_id 胜出
  4. 被覆盖的版本不立即丢弃：
     a. 保存到本地版本历史（最多 30 个版本/30 天）
     b. 写入冲突日志（用户可查看）
```

### 决策 2：冲突版本保留在版本历史中

LWW 导致**整个笔记被覆盖**，但被覆盖的版本不会丢失——通过版本历史保留。

```
每次 LWW 冲突发生时:

  1. 裁决胜出的版本 → 写入 notes 表作为当前版本
  2. 裁决失败的版本（被覆盖的版本） → 写入 version_history 表
  3. 在笔记详情页显示一条温和提示:
     "此笔记在 [设备名] 上有冲突版本, 点击查看版本历史"

version_history 表结构:

  CREATE TABLE version_history (
    id BIGSERIAL PRIMARY KEY,
    note_id TEXT NOT NULL REFERENCES notes(id),
    version INT NOT NULL,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    checksum TEXT NOT NULL,
    device_id TEXT NOT NULL,
    captured_at TEXT NOT NULL,    -- 版本捕获时间
    change_id BIGINT,             -- 关联的 ChangeEntry ID
    conflict_source TEXT,         -- 'local_overwrite' | 'remote_overwrite'
    created_at TEXT NOT NULL
  );
```

### 决策 3：时间戳使用设备本地时间 + 服务端校准

LWW 的裁决准确度依赖于时间戳的可靠性。时间戳策略：

```
本地时间戳:
  - 每次保存时记录设备本地 UTC 时间
  - 本地时间戳用于 ChangeEntry 排序

服务端校准:
  - 每次 Push/Pull 交互时, 客户端携带 device_time, 服务端返回 server_time
  - 客户端计算 clock_skew = server_time - (device_time_send + device_time_receive) / 2
  - 如果 |clock_skew| > 5 秒: 标记设备时间异常

冲突裁决时的时间戳选择:
  - 正常情况: 使用客户端记录的时间戳（本地时间）
  - 时钟偏移 > 5 秒: 以服务端接收时间戳为准（Push 响应中的 server_timestamp）
```

### 决策 4：Deleted 视为最高优先级的操作

删除操作在 LWW 中有特殊处理，避免被编辑操作重新"复活"：

```
删除优先级规则:

  场景: 设备 A 删除笔记, 设备 B 离线编辑该笔记

  1. 当设备 B 的编辑变更到达云端时:
     - 云端检测到目标笔记已被删除（is_deleted = true）
     - 如果设备 B 的编辑时间戳 > 删除操作的时间戳:
       → 保持删除状态, 但将编辑版本保存到版本历史
       → 在冲突日志中记录: "笔记已删除, 编辑版本已保存到历史"
     - 如果设备 B 的编辑时间戳 < 删除操作的时间戳:
       → 直接丢弃编辑变更 (时序上编辑发生在删除前)
       → 无冲突提示

  简而言之: 删除操作一旦完成, 其优先级始终高于后续的编辑操作
  (这其实是一种"Tombstone"机制—删除标记不会因旧变更被覆盖)
```

### 决策 5：冲突类型分类处理

| 冲突类型 | 处理方式 | 用户可见性 |
|---------|---------|-----------|
| **内容 LWW**（双方修改正文） | 时间戳较晚版本胜出，较早版本入历史 | 笔记顶部提示 + 版本历史 |
| **标题 LWW**（双方修改标题） | 同上（整条笔记覆盖） | 同上 |
| **删除 vs 编辑** | 删除优先，编辑版本入历史 | 同上 |
| **重命名 vs 编辑**（A 重命名笔记，B 编辑笔记） | 重命名以最新操作为准；编辑内容与最新标题合并 | 可能产生标题与内容不匹配，建议用户查看版本历史 |
| **标签变更冲突**（A 添加标签，B 移除标签） | LWW 裁决，较晚版本覆盖全部标签列表 | 标签列表可能丢失某次变更 |
| **笔记本归属冲突**（A 将笔记移入笔记本 X，B 移入 Y） | LWW 裁决，较晚版本决定最终归属 | 无特别提示 |

## 备选方案

### 备选 A：使用 CRDT（Conflict-Free Replicated Data Types）

CRDT 是 LWW 的替代方案，可以在字段级别自动合并冲突，不丢失任何用户的变更。

| 对比项 | LWW (选用) | CRDT (备选 A) |
|--------|-----------|--------------|
| 实现复杂度 | 低 — 只需比较时间戳 | 高 — 需要 CRDT 数据结构（RGA / LWW-Register / CmRDT） |
| 数据保留 | 整个笔记被覆盖，但版本历史保留 | 字段级别合并，理论上不丢失任何字符 |
| 同步协议 | 简单 — 直接传递整条笔记 | 复杂 — 需要传递 CRDT 操作（op-based CRDT）或完整状态（state-based CRDT） |
| 带宽消耗 | 低 — 仅传输变更后的 Note 状态 | 中 — 需传输 CRDT 元数据（版本向量、操作日志等） |
| 存储开销 | 低 — 单版本 + 版本历史 | 高 — 需存储 CRDT 元数据 |
| 时钟依赖 | 是 — 依赖时间戳精度 | 否 — CRDT 不依赖物理时钟 |
| 第三方库成熟度 | 无依赖 | 需选型（Automerge, Yjs, Loro） |
| 三端一致性 | 各端自实现 | 需同一套 CRDT 库的三端适配版本 |
| **MVP 适用性** | **高 — 适合 MVP** | 低 — 工程复杂度阻碍快速交付 |

**备选 A 推迟理由**：

CRDT 虽然是更好的最终一致性方案，但引入 CRDT 会带来以下 MVP 阶段难以承担的成本：

1. **三端 CRDT 库适配**：Automerge/Yjs 主要面向 JS/Web 生态，iOS 的 Swift CRDT 实现和 Android 的 Kotlin CRDT 实现需要自行封装或等待社区成熟
2. **同步协议重新设计**：CRDT 需要传递操作（Insert/Delete）而非状态快照，上下游逻辑全部重建
3. **存储模型变更**：CRDT 的底层存储（如 RGA 的有序列表）与普通 SQLite schema 差异大
4. **测试矩阵翻倍**：CRDT 的合并正确性需要大规模随机测试（随机并发编辑验证）

CRDT 作为 P2 增强功能保留在功能清单中（MF-F-208），计划在 v2 或 v2.5 引入。

### 备选 B：合并策略（Three-Way Merge）

类似 Git 的文件合并方式，需要三个版本：本地版本、远程版本、共同祖先版本。

**否决理由**：
- Three-Way Merge 对文本（行级别）有效，但对笔记的 Markdown 内容不一定总能产生有意义的合并结果
- 自动合并可能会产生语法错误的 Markdown（如列表结构被破坏）
- 如果需要人工介入解决冲突，就违背了 MindFlow "零摩擦记录"的产品定位
- Three-Way Merge 的实现复杂度做对（处理所有边缘情况）不亚于 CRDT

### 备选 C：手动冲突解决（弹出冲突窗口）

在冲突发生时，弹出界面让用户手动选择保留哪个版本。

**否决理由**：
- 与 PRD "同步对用户透明 — 不打断工作流" 的定位直接冲突
- 用户故事 E-05-04 明确要求"不弹出让用户手动解决的界面"
- 个人笔记场景中，冲突频次极低（用户同时使用多台设备编辑同一条笔记的窗口期很窄），为此增加复杂的 UI 流程不合理

## 后果

### 正面后果

1. **实现成本极低**：LWW 的核心逻辑约 50 行代码即可实现。各端自行实现，无第三方依赖
2. **用户无感知**：冲突自动解决，不打断用户工作流
3. **零 UI 负担**：不需要冲突解决的 UI 交互设计
4. **同步协议简单**：Push/Pull 传输整条笔记，不需要感知内部细节
5. **天然幂等**：重复同步同一变更不会产生副作用
6. **版本历史保留安全网**：即使用户发现内容被错误覆盖，也能在版本历史中找回

### 负面后果

1. **字段级变更丢失**：这是最严重的代价。示例——用户 A 修改标题，用户 B 同时修改正文。LWW 会导致较晚保存的一方覆盖较早一方的所有变更，包括标题和正文。如果时间窗口刚好相反，两方的变更都被保留
2. **时钟偏移风险**：设备时间不准可能导致 W-LWW（Wrong-Writer-Wins），时间较早的设备变更被错误覆盖。通过服务端校准 + > 5 秒偏移降级策略缓解
3. **标签变更的原子性问题**：Note 的 TagPaths 是集合字段。LWW 将整个标签列表视为原子，A 添加标签 + B 移除标签会导致 A 或 B 的变更丢失。使用集合化的变更记录可缓解但未解决
4. **用户信任感**：一旦用户意识到"我的编辑被另一个设备覆盖了"，即使可以恢复，也会降低对系统可靠性的信任

### 缓解措施

针对负面后果，我们采取以下缓解措施：

| 风险 | 减轻措施 | 紧急响应 |
|------|---------|---------|
| 字段级变更丢失 | 版本历史保留被覆盖版本 | 用户可手动回滚 |
| 时钟偏移导致 W-LWW | 服务端时间校准 + 偏移检测 | 冲突日志可追踪 |
| 标签变更丢失 | 冲突版本保留全部标签状态 | 版本历史可查看 |
| 用户信任缺失 | 温和提示 + 一键恢复入口 | CS 团队支持 |

### 升级路径：从 LWW 到 CRDT

LWW 不影响后续向 CRDT 迁移。迁移路径：

```mermaid
flowchart LR
    subgraph Phase1 [MVP: LWW at Note Level]
        LWW[整篇笔记 LWW\nChangeEntry 记录整篇]
    end

    subgraph Phase2 [v2: CRDT at Content Level]
        LWW2[LWW 保留用于元数据\n标题/标签/笔记本归属]
        CRDT[CRDT 用于正文内容\n基于 Yjs/Automerge]
    end

    subgraph Phase3 [v2.5+: Fine-grained CRDT]
        CRDT2[全字段 CRDT\n正文字段级合并\n元数据自动合并]
        LLM[可选: LLM 辅助语义合并]
    end

    Phase1 -->|1. 正文存储改为 CRDT 数据结构| Phase2
    Phase2 -->|2. 渐进替换元数据 LWW| Phase3

    note for Phase1 "当前阶段\n同步协议传输整篇 Note"
    note for Phase2 "增加 CRDT 字段\n同步协议兼容旧版\n支持混合模式"
    note for Phase3 "全 CRDT\n同步协议升级\n旧版客户端降级为 LWW"
```

**具体迁移关键点**：

1. **存储兼容**：在 v2 迁移时，Note 表的 content 字段旁增加 `crdt_buffer`（JSON 字段），存储 CRDT 的二进制状态快照。旧版本客户端只会读写 `content` 字段，新版本同时读写 `content` 和 `crdt_buffer`
2. **协议兼容**：ChangeEntry 的 payload 中增加可选字段 `crdt_ops`。旧版本不解析此字段（忽略），新版本解析此字段并合并
3. **冲突标记**：在冲突标记中增加 `resolution_strategy` 字段，描述本次冲突使用哪种策略解决（"lww" / "crdt" / "mixed"），便于调试和指标监控

## 冲突处理的纪律规则

为保障 LWW 在 MVP 阶段运行可靠，以下规则必须遵守：

1. **每次 Note 保存必须自增 version** — 依赖 version 检测冲突，不允许跳过
2. **每次 Note 保存必须更新 checksum** — 用于数据完整性校验，防止传输损坏
3. **每个 ChangeEntry 必须携带 device_id** — 冲突日志追踪的唯一标识
4. **device_id 在设备注册时生成并持久化** — 不因 App 重装而改变（使用 Keychain / 持久化存储）
5. **服务端 Push 处理必须按 device_id 分片写入** — 避免多设备并发写入的锁竞争
6. **冲突日志必须记录成功和失败双方** — 用于后期问题排查
7. **版本历史中必须包含冲突来源标记** — 区分"用户主动保存"和"冲突覆盖保存"
